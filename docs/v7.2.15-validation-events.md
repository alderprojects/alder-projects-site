# v7.2.15 — analytics validation events

This release adds the missing trust-and-conversion events the launch gates
need before the November search peak. All events are sent through GA4 via
`window.gtag` (no-op when the gtag stub isn't loaded).

Events are defined in `src/lib/analytics.ts`. Result-page events are
covered separately in `trackResultPageEvent` (see the same file).

## Event reference

| Event name | Where it fires | Why it matters |
| --- | --- | --- |
| `guide_view` | `PageViewEvent` mounted on every `/guides/[slug]` via `TopicGuide` | Top of the guide funnel. Confirms the guide actually rendered for the visitor (vs SSR errored or 404'd silently). |
| `topic_view` | `PageViewEvent` mounted on every `/smart-cart/topic/[slug]` | Top of the cart funnel for paid + organic landings. Distinct from `guide_view` because the topic page is the buyer-intent surface. |
| `smart_cart_cta_click` | `SmartCartGuideCTA` (top / inline / hero variants) | Captures every guide CTA click before the modal opens, so we can compare placements (top vs inline vs hero) for conversion lift. `placement` field discriminates. |
| `curation_modal_open` | `CurationModal` global click handler | Fires whenever any `[data-curation-modal-open]` button opens the modal. `source_path` and `source_component` describe which surface fired it. |
| `curation_modal_prefilled` | `CurationModal` global click handler, only when scope or scenario was passed via data attrs | Confirms the prefill plumbing works end-to-end. If `topic_view` fires but `curation_modal_prefilled` never fires for that scope, the CTA is misconfigured. |
| `cart_start` | covered by `trackSmartCartCurationStep1Completed` (existing) | Buyer completed step 1 of the modal — has scope, scenario, and address. Still funnel-relevant; use the existing event. |
| `cart_result_view` | `trackSmartCartResultViewed` fired from result-page mount (existing) | Confirms the buyer actually saw their cart after checkout. Pre-existing helper. |
| `checkout_start` | covered by the durable buyer-event log (`src/lib/buyer-events.ts`) | Server-side log captures intent-to-checkout. |
| `purchase` | `trackSmartCartPurchased` (existing) | Smart Cart purchase confirmation. Pre-existing helper. |
| `route_out_click` | `WhenToCallAPro` row clicks (existing) | Surfaces the buyers Smart Cart correctly told to leave. Health signal — too low means we're not telling people to leave; too high means our scopes are mis-targeted. |
| `bridge_module_click` | `SmartCartBridge` primary CTA (existing) | Measures whether the contractor-page bridge actually pulls buyers into the cart. v7.2.15 also covers the contractor secondary CTA via `contractor_continue_click`. |
| `contractor_continue_click` | `SmartCartBridge` secondary contractor CTA | Distinguishes "bridge fired the cart" from "bridge fired the pro" so we can read demand split on the basement page. |

## Standard event envelope

Where applicable, every event includes:

- `topic` / `topic_id`
- `scope_variant_id`
- `scenario`
- `source_path` (browser pathname at time of fire)
- `source_component` (the React component that triggered the event)
- `cart_id` (when the event happens after cart creation)
- timestamp (auto-attached by GA4)

No PII. No buyer email or address ever leaves the durable buyer-event log
into GA4.

## Expected interpretation

- **Healthy funnel** for window weatherization (post-Front-Porch-Forum push):
  `guide_view` → `topic_view` → `smart_cart_cta_click` → `curation_modal_open` → `curation_modal_prefilled` → `cart_start` → `purchase` →
  `cart_result_view`. If `curation_modal_prefilled` never fires while
  `curation_modal_open` does, a CTA is dropping its data attributes.
- **Healthy route-out** signal: a small but non-zero `route_out_click` rate on
  basement and window scopes. Zero clicks across a week of traffic = the
  route-out section isn't visible enough.
- **Healthy bridge split**: on `/basement-finishing-vermont`, expect roughly
  60–80% of bridge clicks to hit `bridge_module_click` (cart) vs
  `contractor_continue_click` (pro). Strongly skewed numbers in either
  direction mean the bridge copy is mis-routing intent.

## Where things are NOT instrumented yet

- Picker example-card change: when the visitor toggles the picker on
  `/smart-cart`, we don't fire a "picker_select" event. Acceptable for
  v7.2.15; revisit if the picker becomes a conversion choke point.
- Why-$19.99 module impressions: not tracked. The module is a static block
  with a tested conversion lift in pilots; not worth a per-impression event.

## Validation steps

To verify these events fire in production:

1. Open `/guides/window-film-vs-replacement-vermont` in an incognito tab
   with the GA4 DebugView open. Confirm `guide_view`.
2. Click the top-of-guide CTA. Confirm `smart_cart_cta_click` →
   `curation_modal_open` → `curation_modal_prefilled` (in that order)
   with matching `topic`/`scope_variant_id`/`scenario`.
3. Repeat on `/smart-cart/topic/window-weatherization-vermont`. Confirm
   `topic_view` and the same modal events.
4. Open `/window-replacement-vermont` and click the bridge CTA. Confirm
   `bridge_module_click` with `topic_slug=window-weatherization-vermont`.
5. Open `/basement-finishing-vermont` and click the secondary contractor
   CTA. Confirm `contractor_continue_click` with
   `trade=basement_waterproofing`.
